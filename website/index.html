Durable agent loops in Rust

A crash-safe foundation for LLM agents — mid-turn cancellation with balanced tool-use history, append-only journals that persist and replay, single-writer-per-thread leases, and a durable gRPC + Postgres/SQLite serving host. A Temporal-grade backend for agent loops.

Open-source under MIT · developed in the open and hardening toward production. Not yet battle-tested — expect pre-1.0 breaking changes.

Built for the hard parts

The agent loop is easy to prototype and brutal to make reliable. agent-sdk owns the harness so your tools and prompts stay simple.

🔒

Crash-safe durable execution

Append-only event journal per thread, write-ahead tool-execution intents, and state checkpoints at every turn boundary. Cancel mid-turn and the SDK keeps tool_use / tool_result history balanced.

🌐

Multi-provider LLMs

One LlmProvider trait, many backends: Anthropic, OpenAI, Gemini, Vertex AI, and Cloudflare — each behind a Cargo feature, with streaming and extended-thinking support.

🔌

Model Context Protocol

Connect MCP servers over local stdio subprocesses or remote streamable-HTTP / SSE. Discover tools and resources, then register them straight into your ToolRegistry. feature = "mcp"

🪓

Typed tools & tiers

Implement TypedTool for validated, deserialized input — malformed args become a self-correction result, not a crash. Mark a tool Confirm for human-in-the-loop, or derive it with #[derive(Tool)].

🧮

Structured output

Get schema-valid JSON back from any provider — native JSON mode where supported, or a forced respond tool as fallback — validated and re-prompted on mismatch.

🏗️

Durable serving host

A gRPC service host backed by Postgres or SQLite: task journal, thread & message projections, turn attempts, checkpoints, transactional outbox, and lease-based workers that recover after a crash.

Your first agent

Build the loop, register a typed tool, and ask a question. Persistence, cancellation, and tool dispatch are handled for you.

src/main.rs 
use agent_sdk::prelude::*;
use agent_sdk::ThreadId;
use serde::{Deserialize, Serialize};
use serde_json::json;

// A typed tool: input is validated before `execute` runs.
#[derive(Debug, Serialize, Deserialize)]
struct WeatherArgs { city: String }

struct WeatherTool;

impl TypedTool<()> for WeatherTool {
    type Input = WeatherArgs;

    fn name(&self) -> &'static str { "get_weather" }
    fn display_name(&self) -> &'static str { "Weather" }
    fn description(&self) -> &'static str {
        "Get the current weather for a city."
    }
    fn input_schema(&self) -> serde_json::Value {
        json!({
            "type": "object",
            "properties": { "city": { "type": "string" } },
            "required": ["city"]
        })
    }
    fn tier(&self) -> ToolTier { ToolTier::Observe }

    async fn execute(
        &self,
        _ctx: &ToolContext<()>,
        input: WeatherArgs,
    ) -> anyhow::Result<ToolResult> {
        Ok(ToolResult::success(format!(
            "{}: 18°C, light rain", input.city
        )))
    }
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let mut tools = ToolRegistry::new();
    tools.register_typed(WeatherTool);

    // Reads ANTHROPIC_API_KEY from the environment.
    let agent = builder::<()>()
        .provider(AnthropicProvider::try_from_env()?)
        .tools(tools)
        .build();

    let answer = agent
        .ask(ThreadId::new(), "What's the weather in Lisbon?")
        .await?;

    println!("{answer}");
    Ok(())
}

Prefer full control? Use agent.run_turn(..) for the streaming run-loop, or agent.ask(thread_id, input) for a single turn when full control over application context or cancellation isn't needed.

The durability story

Three storage layers and a single-writer-per-thread contract turn a fragile loop into one that survives process restarts.

Append-only journal
Every turn writes an ordered AgentEvent stream that replays without gaps or duplicates — even with a producer committing during replay.
Write-ahead tool intents
Tool intent is recorded before execution and the result cached after. Crash in between? The retry is idempotent — the result is already there.
Leases & heartbeats
One writer per thread, enforced by leases. If a worker vanishes, expiry sweeps requeue its work — no orphaned turns, no double-counting.

Honest about maturity: the local agent loop and the durable journal are well-tested (including seeded deterministic-simulation fault injection), while transports and multi-instance operation are still hardening. This is a 0.9 technical preview.
